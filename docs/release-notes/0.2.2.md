# 0.2.2

> Shipped 2026-06-09.

The engine grows a spine. This span turns a single-canvas homepage engine into a **layered runtime**:
`@field-ui/core` is purified to zero DOM behind an injected `FieldHost`, a new `@field-ui/platform`
package carries the six registries and frame scheduler, and the field learns to render *over and under*
content. It also carries the naming lineage forward ‚Äî `forces-ui ‚Üí field-ui` ‚Äî and the first npm
publications under the `@field-ui` scope. (Folds in the untagged 0.2.1.)

> Draft release notes ‚Äî paste as the GitHub release body when `v0.2.2` is cut. This file does not itself
> bump versions or tag.

## Added

- **`@field-ui/platform` ‚Äî a new layer between core and the adapters.** Six registries under a
  `createFieldPlatform(root)` read‚Üíwrite coordinator: **Measurement** (frame-stable geometry snapshots,
  Shadow-DOM-safe), **State** (typed element state), **Feedback** (state ‚Üí CSS vars +
  thresholded/hysteretic/debounced events), plus **Relationship**, **VisualBinding**, and **Overlay**
  registries. An explicit six-phase `FrameScheduler`, platform **lint** guardrails, and `bindData()` ‚Äî
  records participate in field behavior. Strict dep direction: `platform ‚Üí core`, core never depends up.
- **Renderer-agnostic core via an injected `FieldHost`.** The engine no longer reaches for the DOM
  itself; all environment access goes through a host, opening the door to non-DOM renderers.
  `<field-root>` is flipped onto the platform runtime by default (behind the D-series migration).
- **Field Surfaces ‚Äî placement as a first-class concept.** `setOverlay(mode)` composites the live field
  *in front of* content on a second canvas; combined with the underlay it reads as *immersive*.
  `OverlayMode` / `overlayCanvas` added to `FieldOptions` and `FieldHandle`, drawn without a second
  engine or scan (core stays DOM-free).
- **The Field Agent Consumption Model ships.** Four consumer cells with pure, DOM-free decision cores:
  sink hold‚Üírelease (conserved capture‚Üíhold‚Üírelease cycle), element dock, `warp` relocate, and element
  emit ‚Äî plus capture/release events. Declarative visual binding (`data-field-visual-for`) and `scan()`.
- **Natural Fields.** The four-field reframe (Gravity‚Üíimportance, Electromagnetic‚Üípolarity,
  Strong‚Üíbinding, Weak‚Üítransformation) with a classification model, canonical doc, `/docs/natural-fields`,
  and natural-field badges + filtering in the Lab.
- **Recipe system.** A first-class `FieldRecipe` model, the catalog expanded to **64 recipes across four
  tiers**, a recipe runtime (compile + apply + metric library), a live Platform Inspector and recipe
  inspector, and a recipe gallery.
- **Diagnostics + render modes.** On-canvas diagnostic overlays ‚Äî topology, inspector, causality,
  prediction ‚Äî an energy/potential/probe/heatmap diagnostics module, and PNG/SVG field export.
- **The developer portal (`/docs`)** ‚Äî getting started, concepts, framework guides, a catalog-driven API
  reference, live force demos, a first-project tutorial, and a Troubleshooting page. Plus `apps/starter`,
  an in-repo Vite starter.
- **The research family.** An 8-paper arXiv-style set published at `/research`, rendered with KaTeX, plus
  a `/writings` datastore and the `/eli5` plain-language second homepage.
- **`@field-ui/vanilla`** ‚Äî a framework-free TypeScript wrapper package.
- **The 0.x public API freeze**, enforced in CI (14 entries at cut).

## Changed

- **Naming lineage carried forward: `forces-ui ‚Üí field-ui`.** A hard rename with a compatibility-alias
  window (`--forces-* ‚Üí --field-*`, `forces:* ‚Üí field:*`) ‚Äî repo, domain, canonical classes, and docs
  swept field-first. Six canonical tokens renamed to functional terms.
- **The core is published as `@field-ui/core`** (the unscoped `field-ui` name is blocked on npm by a
  similar package). All internal deps and `import ‚Ä¶ from 'field-ui'` specifiers resolve to the scoped
  name; API surface otherwise unchanged.
- **`--field-density` is the primary feedback variable;** the site leads with the `data-*` attribute
  contract, not physics. Recipe tiers renamed to a complexity ladder (applied/systems/operational).
- **DOM-free core enforced** ‚Äî legacy DOM glue quarantined behind a boundary guard; `@field-ui/core`
  imports zero DOM.
- Expanded the `react`, `elements`, and `vanilla` READMEs with full options/methods tables and the
  `data-body` vocabulary.

## Fixed

- **Real relationship + confidence math** ‚Äî the platform computed actual relationship resolution instead
  of hardcoding `relResolved = relTotal`, and stopped fabricating confidence from mere relationship
  presence.
- **`memory` reclassified** as a semantic metric, not a physical natural force; risk is supplied-only
  (never defaulted to 0).
- **Live attribute reactivity + full field teardown** in `<field-root>`; `threads()` proxied onto the
  element; SiteNav toggles and the home field survive navigation, drag, and resize without resetting.
- **Correct force count** (34‚Üí35) across Paper 1 and site copy; catalog-driven counts throughout the docs.

## Published

`@field-ui/core`, `@field-ui/platform`, `@field-ui/elements`, `@field-ui/react`, and `@field-ui/vanilla`
are live on npm, plus `@field-ui/kit` / `@field-ui/field-ui` for one-install consumption. A
**tag-triggered provenance release workflow** (npm provenance via GitHub OIDC) becomes the path for all
future releases.
