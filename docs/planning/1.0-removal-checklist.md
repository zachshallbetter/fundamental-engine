# 1.0 deprecation-removal checklist — the human cut

> **Status: planning (G3 prep — this executes NOTHING).** This is the file-level inventory of exactly
> what gets deleted at the `1.0` cut, so the removal is mechanical and low-risk. The **decision** to
> remove these (kept-vs-deprecated split, `1.0` target) is recorded in
> [`../canonical/deprecation-plan.md`](../canonical/deprecation-plan.md) and the "Deprecation & removal
> policy" section of [`../canonical/api-stability.md`](../canonical/api-stability.md). The **consumer**
> upgrade guide is [`../analysis/migration-0.x-to-1.0.md`](../analysis/migration-0.x-to-1.0.md). The actual removal +
> `v1.0.0` release is a **separate, human-initiated** step — nothing here is run by preparing it.
>
> Line numbers are against `main` at the time of writing; re-grep before cutting (the repo moves).

## How to use this

At the 1.0 cut, an engineer works down §1–§4 below: delete the cited symbol/line, update the cited
tests, then run the full gate (`pnpm typecheck build test check:api check:dist check:docs check:cem`).
Each item lists **(a)** the source site(s), **(b)** the tests that will fail and need updating, and
**(c)** the cross-plane (Swift/Android) status. **The alias surface is smaller than the policy list
implies** — most of the rename-era surface was already removed at `0.7.0` (see §5), and the two
DOM-event/CSS-var aliases have **no native-port equivalent** (see §6).

## 1. `forces:*` event alias — REMOVE at 1.0

The engine dispatches `forces:captured` / `forces:released` / `forces:relocated` alongside the canonical
`field:*` events. Remove the `forces:*` dispatch (and its dev-warn).

**(a) Source — `packages/core/src/engine/field.ts`, `fireCaptureEvent` (~L1043–1051):**

- **L1045** — delete the `forces:` dispatch:
  `el.dispatchEvent(new CustomEvent('forces:' + name, { bubbles: true, composed: true, detail }));`
- **L1046–1050** — delete the `devWarnDeprecated('forces:' + name, …)` call (the whole block).
- **L1038–1042** — update the leading comment (it describes the alias + the removal-at-1.0 warn).
- Keep the `field:` dispatch on **L1044** unchanged.
- `devWarnDeprecated` (imported at **L68** from `../contracts/guards.ts`) may still be used by other
  aliases at cut time (it is generic) — **do not delete the helper**; only drop the call site. Verify
  `grep -rn devWarnDeprecated packages/core/src` after the cut; if this was its last caller, the import
  on L68 becomes unused and should be dropped too.

**(b) Tests to update:**

- `packages/core/src/engine/frame-harness.test.ts` — L74 (`forces:captured` alias fired), the whole
  dev-warn test at **L81–107** (`the forces:* alias fires a dev-only deprecation warn…`), L161
  (`['field:captured','forces:captured','field:released','forces:released']` expectation), L218
  (`forces:relocated` alias). These assert the alias *exists*; invert them to assert it is **gone** (no
  `forces:*` in `dock.events`), or delete the alias-specific cases.
- `packages/core/src/contracts/guards.test.ts` — L135–164 exercise `devWarnDeprecated` using
  `forces:captured`/`forces:released` as sample ids. The helper itself is generic and stays; if it keeps
  other callers, these tests can keep their sample strings or switch to a surviving alias id. If
  `devWarnDeprecated` becomes caller-less, delete this test block with the helper.
- `packages/core/src/engine/frame-harness.ts` — L63, L72 are doc comments naming `forces:*` as a recorded
  event type; update the prose (no behavior).

**(c) Cross-plane:** **none.** Neither Swift nor Android carries a `forces:*` event alias — verified:
`grep -rn "forces:captured\|forces:released\|forces:relocated" swift/ android/` returns **zero** hits
(the `forces:` matches in the ports are the Swift/Kotlin `forces:` *parameter label* on
`StepInput`/`ForceRegistry`, unrelated). The ports emit capture/release through their own APIs
(`CaptureEvent` in `swift/.../Reactions.swift`), never under a `forces:` name. This removal is
**JS/DOM-plane only.**

## 2. `--mass` CSS-var alias — REMOVE at 1.0

`--mass` is written alongside the canonical `--load` on every sink body. Stop writing it. **This has
more sites than the policy's two — four functional/list sites plus comments:**

**(a) Source — functional write sites (delete the `--mass` write):**

- `packages/core/src/engine/feedback-sink.ts` **L47** — `el.style.setProperty('--mass', loadStr);` (the
  internal default sink). Delete this line; keep the `--load` write on L46.
- `packages/elements/src/platform-runtime.ts` **L123** — drop `'--mass': f3(ch.load)` from the
  `platform.feedback.set(el, { '--load': …, '--mass': … })` call (leave `--load`).

**(a′) Source — list/cleanup sites (remove `'--mass'` from the array):**

- `packages/core/src/engine/field.ts` **L680** — the `clearWriteback` var-list
  `['--d','--field-density','--load','--mass','--entropy',…]`. Remove `'--mass'` so the write-back
  cleanup no longer clears it.
- `packages/dom/src/visual-bindings.ts` **L75** — `MIRRORED_CHANNELS` (the list `bindData` uses to
  mirror a source body's feedback vars onto a target). Remove `'--mass'` so the visual-binding mirror
  stops copying it.

**(a″) Comment-only references (update prose, no behavior):**
`accretion.ts:61`, `core/types.ts:788`, `field.ts:1627`, `feedback-sink.ts:28`,
`config/forces.config.ts:148`, `dom/visual-bindings.ts:19`, `elements/platform-runtime.ts:122` — each
mentions `--load` / `--mass` in a doc comment; drop the `/ --mass` half.

**(b) Tests to update:**

- `packages/core/src/engine/feedback-sink.test.ts` — **L49** (`load writes --load then the --mass
  back-compat alias`), **L54** (`['--mass','1.000']` expectation), **L70** and **L131** (var-list
  expectations that include `--mass`). Drop `--mass` from the expected keys; the L49 test's premise
  disappears (rename/retarget it to assert `--mass` is **not** written).
- `packages/dom/src/visual-mirror.test.ts` — **L53, L56** seed and assert a mirrored `--mass`. Retarget
  to `--load` (or assert `--mass` is not mirrored).
- `packages/elements/src/platform-runtime.feedback.test.ts` — **L34** (`load routes to --load + --mass`)
  and **L40** (`assert.deepEqual(flat, { '--load':…, '--mass':…, … })`). Drop the `--mass` key.

**(c) Cross-plane:** **none.** `grep -rn "\-\-mass" swift/ android/` returns **zero** hits — neither
port ever wrote `--mass` (the Swift port renamed the field itself: `accreted // captured load (was
mass, §21.2)`). CSS vars are a DOM-plane concern; the native feedback sinks use their own channel names.
**JS/DOM-plane only.**

> Do **NOT** touch `--load`, `--d`, or `--field-density` — those are kept permanently (canonical dual
> naming; see the deprecation plan). The `--entropy`/`--coherence`/`--temperature` bare names in the
> same arrays are experimental measured-thermo channels, **not** aliases — leave them.

## 3. `@fundamental-engine/platform` alias package — STOP PUBLISHING at 1.0

A thin package that re-exports `@fundamental-engine/dom` so pre-0.7.0 imports keep working.

**(a) Source — the package dir `packages/platform/`:**

- `packages/platform/src/index.ts` — the entire alias module: `export * from '@fundamental-engine/dom';`
  (**L16**) plus the one-time dev-only import `console.warn` (**L22–30**). Remove the package.
- `packages/platform/package.json` — `"name": "@fundamental-engine/platform"`, description already reads
  `"DEPRECATED — renamed to @fundamental-engine/dom…"`. Remove from the publish set.
- `packages/platform/README.md`, `LICENSE`, `tsconfig.json`, `dist/` — remove with the dir.
- **Publishing:** `packages/platform` is one of the **seven** packages in the publish filter
  (`@fundamental-engine/*`). After removal, six packages publish (core, dom, elements, react, vanilla,
  three). Confirm `RELEASING.md` / `PUBLISHING.md` no longer list `platform` and that nothing in
  `.github/workflows/release.yml` names it explicitly (it uses the `@fundamental-engine/*` filter, so
  removing the dir is sufficient).
- **Deprecate on npm** (optional, recommended): `npm deprecate @fundamental-engine/platform "renamed to
  @fundamental-engine/dom"` so the last published version points migrators at `dom`.

**(b) Tests to update:** none in `packages/platform/src` (`ls` shows no `*.test.ts`). Grep confirms the
only tracked references to the string `@fundamental-engine/platform` outside the package itself are
prose: `packages/core/src/contracts/guards.ts` **L94** (a doc comment listing the aliases) and
`apps/site/public/llms-full.txt` (a generated corpus — regenerates). No test imports the alias package.
The frozen-API gate does **not** name `platform` (verified: `scripts/api-surface.data.mjs` references
`@fundamental-engine/dom`, never `/platform`), so `check:api` is unaffected.

**(c) Cross-plane:** **none.** The Swift/Android ports have `FundamentalPlatform` / `:fundamental-platform`
as their *primary* platform module — that is the real layer, **not** an alias, and it is **not** renamed
or removed (the JS `platform → dom` rename was a JS-package decision that the ports never mirrored). Do
not touch the native platform modules. **JS-plane only.**

## 4. `<field-field>` custom element — DECIDE at 1.0 (remove OR bless)

`<field-field>` is registered as a secondary tag for the singleton field, alongside the canonical
`<field-root>`.

**(a) Source — `packages/elements/src/index.ts`:**

- **L702–704** — `if (… && !customElements.get('field-field')) { customElements.define('field-field',
  FieldField); }`. To remove: delete this registration (keep `field-root` on L713–715).
- **L719** — the `HTMLElementTagNameMap` entry `'field-field': FieldField;` (drop if removing).
- **L26** — doc comment naming both tags (update if removing).
- Note the class shape: `FieldField` is the base; `FieldRoot extends FieldField` (**L711**). Removing the
  `field-field` *tag registration* does not touch the `FieldField` **class**, which stays as the base and
  is a frozen export (do not remove the class).

**Caveat — this one is a MAINTAINER DECISION, not an automatic removal.** The deprecation plan
(row #3) explicitly says the maintainer may **bless `<field-field>` as a permanent secondary
registration** instead of removing it, to be decided at the cut. `api-stability.md` currently documents
`<field-root>` as "also registered as `<field-field>`". **Confirm the call before editing** — if blessed
permanent, this item is a no-op and the api-stability wording stays; if removed, update that wording too.

**(b) Tests to update (only if removing):** grep `packages/elements/src` for `field-field` registration
assertions and the CEM. The Custom Elements Manifest `packages/elements/custom-elements.json` (gated by
`check:cem`, regenerated via `pnpm gen:cem`) currently lists `field-field` — regenerate after removing
the tag. Also re-grep `apps/`, `docs/`, and `e2e/` for `<field-field>` usage.

**(c) Cross-plane:** n/a — custom-element tags are DOM-plane only.

## 6. Notes / findings — where the policy is imprecise

Cross-checked the checklist against the code; two corrections to how the policy reads:

1. **The `forces:*` and `--mass` aliases are JS/DOM-plane only — no cross-plane mirror.** The Swift and
   Android ports carry **neither** (`grep` for the event names and the var returns zero hits in `swift/`
   and `android/`). The policy docs do not claim they do, but a reader might assume "fix in all planes"
   per the repo's engine-fix rule — that rule does **not** apply here: there is nothing to remove in the
   ports. Documented in each item's (c).

2. **`--mass` has four functional/list sites, not the two the policy names.** The deprecation plan cites
   `feedback-sink.ts` + `platform-runtime.ts` (the write sites). It omits the `clearWriteback` var-list
   in `field.ts:680` and `MIRRORED_CHANNELS` in `dom/visual-bindings.ts:75`, both of which also carry
   `'--mass'` and must be updated for a clean removal (otherwise the mirror/cleanup still references a
   var nothing writes — harmless but stale). Captured in §2(a′).

Everything else in the policy checks out against the code (line references confirmed).

## Post-removal gate

After the cut, the standard sweep plus the two easy-to-miss CI gates:

```
pnpm typecheck && pnpm build && pnpm test && \
pnpm check:api && pnpm check:dist && pnpm check:readme && \
pnpm check:recipes && pnpm check:cem && pnpm check:docs && pnpm check:links
```

Then the release checklist in `RELEASING.md` (bump `FIELD_VERSION` in `packages/core/src/version.ts` to
`1.0.0` in lockstep; revert `apps/site` / `apps/starter` package versions after the `@fundamental-engine/*`
bump). The `v1.0.0` tag is the human-initiated trigger — **not** part of preparing this checklist.
