# F1.1 — FieldPattern integration audit & mapping

> **Status: audit (grounded against code on `feat/world-kernel`).** The mandated pre-implementation
> audit of the existing `FieldPattern` execution path, and the structural mapping to generic kernel
> declarations. It reaches a **finding that a stop condition applies before kernel code is written** —
> see §Finding. Feeds F1.1 in [`PLAN.md`](PLAN.md) and the kernel-role record
> [`m1.5/06-kernel-role-hypotheses.md`](m1.5/06-kernel-role-hypotheses.md).

## The governing fact
A `FieldPattern` never becomes engine-as-data. `compilePattern` (`packages/core/src/recipes/compile.ts:142`)
produces a `CompiledPattern` whose `bodies` are **stringly `data-*` attribute bags + a token list**
(`compile.ts:21-30, 104-117`) — authoring markup, not `Body` objects, no numeric physics. The only shipped
pattern→engine bridge, `applyPattern` (`packages/dom/src/apply-recipe.ts:128`), drives an already-running
`FieldHandle` through three output setters (`setRender`/`setOverlay`/`setHeatmap`) — **delegation, by
construction.** Bodies reach the engine only via `addBody` (`engine/field.ts:3388`), which even fabricates
a fake DOM element to reuse the scan path (`field.ts:3407-3419`).

## 10-point audit (file:line grounded)
1. **Construction:** `compilePattern → CompiledPattern` (pure); running fields are `createField(canvas, opts)` (`field.ts:211`). No compile→createField path exists. `CompiledPattern` = `{id, recipe, bodies[], relationships[], feedback[], diagnostics[], metrics[], conditions[], render, reducedMotion}` (`compile.ts:104-117`).
2. **State ownership:** scattered across ~20 closures inside `createField` — `store: FieldStore` (`field.ts:257`), `bodies` (`461`), `programmaticEdges` (`330`), `env` (`885`), `frameN` (`480`), `focusLedger` (`521`), grids (`259`). **No central serializable world-state object.**
3. **Identity:** `FieldBodyIdentity {id,namespace?,kind?,host?}` (`types.ts:177`), derived deterministically `cached → identify(el) → DOM id → body-N` (`field.ts:503-513`); relations keyed by `` `${from} ${to} ${type}` `` (`field-snapshot.ts:23`). Clean and host-independent.
4. **Update order:** `frame()` (`field.ts:2825`) — measure every `frameN%6` (`2873`), `frameN%30` scroll resample (`2913`), then `step()` integrator (`integrator.ts:223`). Deterministic given `now`+`rng`; `env.dt` is wall-clock-derived (`2837`) but `===1` on every golden/conformance run (byte-identical there).
5. **Precision/serialization:** live sim f64; render buffer f32 (`types.ts:22-27`). Serializable = `FieldSnapshot` (`types.ts:1327`, versioned by `FIELD_VERSION`), **lossy** (no velocities/particle-ids/registry). Nondeterminism via `Math.random`/`performance.now` defaults + wall-clock `dt`, all injectable.
6. **Invariants:** particle **count** is the one strong invariant (documented, `field-store.ts:5-6`; not runtime-asserted); passport validity enforced at **validation** time (`schema.ts:213-233`), not runtime; `check:api`/contract-coverage gate the public surface, not physics.
7. **Snapshot/replay:** snapshot captures per-body `{id,identity,authority,rect,position,tokens,metrics,dimensions,data?}` + edges + field metrics (`field.ts:3609-3684`). `diff`/`replay` are **pure over two snapshots** (`field-snapshot.ts:27,105`) — read-two-snapshots, **never re-run**. A snapshot **cannot reconstruct a live field**.
8. **Failure modes:** plain `Error` message-encoded (`validatePattern`/`parsePattern` `schema.ts:257`, `applyPattern` `apply-recipe.ts:141`, `addEdge` `field.ts:3473`, missing 2D ctx `field.ts:221`). One typed error exists: `IncompatibleWorldVersion` (F1.0, `world/envelope.ts:59`).
9. **Implicit global/host state:** none at module level; `createField` is self-contained per instance; banned globals routed through `opts.rng`/`opts.now` (`field.ts:765-766`); environment via injected `FieldHost` (`host.ts:39-85`).
10. **Temptation points (field-specific knowledge that would bleed into a generic kernel):** the stringly DOM attribute bag as "entity"; `render`/`overlay`/`heatmap` matter-mode taxonomy as "operations"; `feedback` CSS-var surface as "projection"; the six force channels (linear/thermal/angular/temporal/semantic, `integrator.ts:106-153`) as "dimensions"; the `Env` god-object (`types.ts:453-532`) as "tick context"; **`Force.apply(b,p,env)` code as "operations"**; `frameN%6` cadence as kernel scheduling; DOM-id identity fallback; `addBody`'s fake element. The sharpest: letting `kernelSemantics` silently become "the force-registry version."

## Structural mapping (CompiledPattern → generic kernel declarations)
| FieldPattern construct | Kernel representation | Preserved | Transformed | Omitted |
|---|---|---|---|---|
| `bodies[].tokens` + params | Entity `{identity, tokens, params}` | tokens, identity | stringly `data-*` → typed params (lossy round-trip) | — |
| `FieldBodyIdentity` | Entity identity | all (`{id,ns,kind,host}`) | — | — |
| `relationships[]` | Relation `{from,to,type,strength}` | all (identity-keyed) | — | — |
| `recipe.expected` (`particleCount`, `entropyRange`, `energyDriftMax`) | Invariant declarations | all | — | — |
| `render`/`feedback` | Projection declarations | intent | matter-mode/CSS → generic projection refs | field-specific layer semantics |
| `FieldSnapshot` | State (read-only) | serializable subset | — | velocities, particle ids, registry, grids |
| **force tokens** | Operation **reference** (opaque) | token names | — | **the force math (the law itself)** |

## Finding (the crux — B)
**State: representable as read-only, lossy data. Lawful evolution: NOT declaratively representable.**

The field's laws are **executable code** — each `Force` is `apply(b,p,env): void` mutating velocity/heat/age
(`types.ts:586-604`; `applyAndRecord` `integrator.ts:90-155`), living in a per-field `ForceRegistry`
(`types.ts:644`, built inside `createField`, `field.ts:260`). A `CompiledPattern` carries force **tokens**,
never their math. Live state is closure-scattered and a snapshot is lossy, so it cannot rehydrate a field.

Therefore, to evolve a hosted world, a kernel must **either** retain an opaque `FieldHandle` and delegate
`tick()` back to `frame()`/`step()`, **or** invoke the opaque per-field registry — both are **delegation**.

This trips two declared F1.1 **stop conditions** *before any kernel code is written*:
- "the kernel must retain an opaque `FieldPattern` instance to execute";
- "the kernel merely delegates execution back to the old runtime without representing its state and lawful evolution."

## Determination for F1.1
- **Achievable, non-delegated:** a generic kernel can host, as faithful data, the field's **entities
  (identity + tokens + params), relations, invariants, projection declarations, and read-only snapshot
  state** — with omissions declared, not hidden. This is real generic structure, not a wrapper.
- **Not achievable without delegation:** hosting the field's **lawful evolution** generically. The
  "Operations/Dynamics" kernel elements are, for the field, **opaque code, not declarations.**
- **Finding class (per the F1.1 report options):** *"the field can be represented only with additional
  generic structure"* — specifically, the evolution law needs a **declarative force IR** that does not
  exist today. Not option 1 (K does not genuinely host evolution) and not a flat option 4 (the
  declaration/state representation is genuine structure, not a versioned wrapper).
- **Feeds M1.5-06:** reclassify **Operations** and **Dynamics** from "formal primitive / labeled Dynamics"
  toward **"not currently declarative — opaque code pending a force IR."** This is a real ablation-style
  finding (record 06's open question resolving negatively for the field plane).

## Recommended next step (design fork — needs a call)
Because a *running-field* host would fail the stop conditions, the honest F1.1 build is a **declaration +
state + invariant host with evolution as an explicit opaque operation** (no delegation, no opaque field
instance retained, no field imports in the kernel; a one-way `CompiledPattern → KernelWorld` adapter kept
separate and guarded). That demonstrates the finding concretely. The alternative — designing a
**declarative force IR** so evolution becomes data — is the "additional generic structure" the finding
points to, and is a larger piece of work than F1.1 scoped. See the report accompanying this commit.
